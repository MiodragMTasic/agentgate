AgentGate onboarding explainer

AgentGate, explained like a product instead of a puzzle.

AgentGate is not the model. It is the policy layer in front of the model’s tools. Claude Code, Codex, OpenAI, Anthropic, or MCP can still choose tools, but AgentGate decides whether those tool calls are allowed, denied, or paused for approval.

One question Should this exact tool call be allowed right now?

That is the whole job of AgentGate.

Best fit Agents that can touch real systems

Files, commands, email, deploys, money actions, admin tools, and MCP.

Where AgentGate sits

The easiest mistake is to think AgentGate is “another AI.” It is not. It is the decision point between an agent runtime and the tools that can affect your code, users, money, or production state.

User request

“Check the build, email the beta user, and deploy a preview.”

→
Agent runtime

Claude Code, Codex, OpenAI, Anthropic, or an MCP client decides which tool to call.

→
AgentGate

Checks policy, params, rate limits, budgets, and approval requirements.

→
Real tool

Files, commands, deploys, email, admin APIs, database access, Stripe, Firebase.

Diagram 2

There are really two onboarding paths

The core confusion came from mixing together two valid but different setups. Most users should start with the tandem path on the left.

Path A · recommended first

Claude Code / Codex in tandem

You already use Claude Code or Codex.
You expose risky project actions as MCP tools.
AgentGate sits in front of those MCP tools.
The agent still feels normal to use, but the dangerous calls are filtered.
Use this when:

You want Claude Code or Codex to help with a real project without being able to read secrets, quietly deploy, or run destructive commands.

Path B · lower-level integration

Direct OpenAI / Anthropic SDK wrapping

You already have app code that defines tools in the vendor SDK.
You wrap those tool definitions with AgentGate adapters.
AgentGate enforces policy before the tool executes.
This is ideal for application code, backends, and custom agents.
Use this when:

You are building the agent product itself and already own the direct SDK tool wiring.

Diagram 4

What this would look like for Wanderlust

Here is the concrete version. Do not try to guard everything on day one. Start with the smallest set of actions that creates a meaningful trust boundary.

Start allowed

Helpful, low-risk actions

Read docs, routes, and safe source files
Run pnpm test, pnpm lint, and pnpm build
Inspect safe public trip data
Start denied

Things that cross trust boundaries

Read .env or secret files
Run destructive shell commands
Write to Firebase Admin or Stripe money flows
Start approval-gated

Actions that should pause for a human

Deploy preview or production
Send user-facing outbound email
Create reservation or handoff actions that affect real users
Diagram 5

The first 30 minutes

This is the concrete onboarding sequence a real user should follow. The goal is not “enterprise policy perfection.” The goal is “first useful safety boundary.”

Step 1

Generate the right starter

pnpm dlx @miodragmtasic/agentgate-cli init --runtime claude-code --project-name Wanderlust

This creates a policy file, scenarios file, and a runtime-specific onboarding guide.

Step 2

Rename the sample tools

Replace the starter tool names with your real project actions like read_project_file, run_safe_command, and deploy_preview.

Step 3

Validate the policy

agentgate policy validate
agentgate test

Do this before wiring anything into your live runtime.

Step 4

Put AgentGate in front of MCP tools

For Claude Code and Codex, the cleanest default is usually guarded MCP tools.

Step 5

Let the agent use the guarded tools

The runtime still feels normal, but dangerous calls are filtered by policy.

Step 6

Add more tools later

Only after the first boundary feels trustworthy should you widen the tool surface.
